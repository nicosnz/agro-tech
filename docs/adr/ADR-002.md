# ADR-002: Por qué una arquitectura FSD y no MVC en el frontend

Estado: Aceptada
Fecha: 2026-04-15

## Contexto

El frontend de AgroTech es una SPA construida con React y Vite. Tiene al menos
6 entidades de dominio (bovino, lote, potrero, pesaje, alimentación, tipoAlimento),
varias features compuestas (registrar-pesaje, registrar-alimentacion,
filtrar-ganado-por-estado) y componentes compartidos reutilizables.

Al iniciar el proyecto se evaluó cómo organizar el código fuente del frontend.

## Opciones consideradas

**a) MVC clásico adaptado (carpetas por tipo técnico)**
```
src/
  models/
  views/
  controllers/
  services/
  components/
```
El código se agrupa por su rol técnico: todos los modelos juntos, todos los
componentes juntos, etc.

**b) Feature-Sliced Design — FSD (opción elegida)**
```
src/
  app/          ← inicialización global, providers, layouts
  pages/        ← páginas completas (ganado, pesajes, alimentacion)
  widgets/      ← bloques UI compuestos por features (ganado-table, pesaje-table)
  features/     ← acciones del usuario (registrar-pesaje, filtrar-ganado-por-estado)
  entities/     ← objetos de dominio (bovino, lote, potrero…)
  shared/       ← UI y utilidades sin dominio (modal, pagination, api client)
```
El código se agrupa por feature/dominio con capas con dirección de dependencia
estricta: cada capa solo puede importar de capas inferiores.

## Decisión

Adoptamos la opción (b): Feature-Sliced Design.

## Razones

1. **MVC no encaja con componentes React.** En una SPA con React el componente
   mezcla vista y controlador por diseño. Separar artificialmente `views/` de
   `controllers/` produce duplicación sin ganancia: el componente y su lógica
   terminan separados en carpetas distintas pero son inseparables en la práctica.

2. **Escalado por dominio, no por tipo técnico.** Al agregar la entidad
   `vacunación`, FSD indica exactamente dónde va cada archivo:
   `entities/vacunacion/{api,model,ui}`. En MVC habría que tocar `models/`,
   `components/`, `services/` y `hooks/` simultáneamente sin una guía clara.

3. **Regla de dependencia unidireccional.** FSD prohíbe que una capa inferior
   importe de una superior (`entities` no puede importar de `features`). Esto
   elimina la categoría entera de bugs de dependencias circulares y hace el
   grafo de módulos predecible.

4. **Co-localización de api + model + ui por slice.** Cada entidad lleva sus
   propias llamadas a la API, sus tipos TypeScript y sus componentes juntos.
   En MVC esos tres artefactos viven en tres carpetas distintas, lo que
   obliga a navegar el árbol de directorios para entender una sola feature.

5. **El dominio del proyecto es rico y crecerá.** Las entidades ganaderas
   (bovino, lote, potrero, pesaje, vacunación, alimentación…) son candidatas
   naturales a slices FSD. Agregar nuevas entidades no requiere reorganizar
   código existente.

## Estructura real adoptada

| Capa       | Slices presentes                                                    |
|------------|---------------------------------------------------------------------|
| `pages`    | ganado, pesajes, alimentacion                                       |
| `widgets`  | ganado-table, pesaje-table, alimento-grid                           |
| `features` | registrar-pesaje, registrar-alimentacion, filtrar-ganado-por-estado |
| `entities` | bovino, lote, potrero, pesaje, alimentacion, tipoAlimento           |
| `shared`   | api client, modal, pagination, badge, toast, dataTable, …          |

## Consecuencias

+ Añadir una entidad nueva (ej. `vacunacion`) no requiere tocar capas existentes.
+ La regla de imports hace que los errores de dependencia circular sean
  detectables por linting estático.
+ La carpeta de cada feature es autocontenida: fácil de borrar o mover.
- FSD tiene más convenciones que aprender que una estructura MVC familiar;
  requiere que todo el equipo las respete para que el beneficio se mantenga.
- Algunos frameworks y tutoriales asumen MVC; integrar librerías de terceros
  a veces requiere decidir conscientemente en qué capa van sus wrappers.
