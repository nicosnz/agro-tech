# ADR-001: Por qué monolito modular y no microservicios

Estado: Aceptada
Fecha: 2026-04-15

## Contexto

AgroTech es una plataforma de gestión ganadera desarrollada por un equipo pequeño
(1 persona) en el contexto académico. El sistema tiene dos dominios principales:

- **admin** (Django): gestión de usuarios, configuración del sistema y panel administrativo.
- **api-vaquero** (FastAPI): dominio ganadero — bovinos, lotes, potreros, pesaje,
  vacunación, alimentación y búsqueda (Elasticsearch).

Ambos dominios comparten una única base PostgreSQL y se despliegan juntos mediante
un único `docker-compose.yml` con una red privada interna.

Al diseñar la arquitectura se evaluó si separar los dominios en microservicios
independientes era la opción correcta para este proyecto.

## Opciones consideradas

**a) Microservicios independientes**
Cada dominio tiene su propio repositorio, pipeline CI/CD, base de datos, y se
comunica con los demás exclusivamente por HTTP o mensajería asíncrona.

**b) Monolito modular (opción elegida)**
Un único repositorio con módulos bien delimitados (`admin`, `api-vaquero`, `ui`),
desplegados como contenedores en una red privada compartida, con una base de datos
compartida y un proxy Nginx como único punto de entrada.

## Decisión

Adoptamos la opción (b): monolito modular desplegado con Docker Compose.

## Razones

1. **Equipo pequeño.** Los microservicios están pensados para que equipos
   independientes puedan desplegar sin coordinarse. Con 21 persona el overhead
   de mantener pipelines, registries y versionado de contratos de API separados
   supera cualquier beneficio.

2. **Dominio cohesivo.** Los datos entre `admin` y `api-vaquero` están
   relacionados (usuarios ↔ bovinos, fincas ↔ potreros). Separar esos dominios
   implicaría joins distribuidos o duplicación de datos.

3. **Despliegue único.** `docker-compose up` levanta todo el sistema. Un solo
   comando elimina la complejidad de service discovery, health-check cross-service
   y gestión de versiones entre servicios en producción.

4. **Observabilidad más simple.** Sin tracing distribuido, los logs de un
   `docker-compose logs` son suficientes para diagnosticar cualquier fallo.

5. **Migración incremental posible.** La separación física ya existe: cada módulo
   vive en su propio directorio con su propia imagen Docker. Si un módulo necesita
   escalar de forma independiente en el futuro, puede extraerse a un microservicio
   sin reescribir el dominio.

## Consecuencias

+ Ciclo de desarrollo rápido: un solo repositorio, un solo entorno local.
+ Sin latencia de red para operaciones que cruzan dominios.
+ Despliegue atómico: imposible que `admin` y `api-vaquero` estén en versiones
  incompatibles en producción.
+ Los límites de módulo están en el código, no forzados por la red; requiere
  disciplina para no crear dependencias cruzadas directas.
- Si un módulo tiene un pico de carga extremo, escala todo el stack junto.
- El despliegue es todo-o-nada; un error en un módulo puede afectar al resto
  si no se aíslan correctamente los health-checks de Docker.
