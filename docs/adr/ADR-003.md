# ADR-003: Por qué React y no Angular o VanillaJS

Estado: Aceptada
Fecha: 2026-04-15

## Contexto

AgroTech necesita una SPA que consuma la API REST de `api-vaquero` y presente
vistas de gestión ganadera (listados, formularios, búsqueda, estadísticas).
El equipo es de 1 persona con conocimiento previo de JavaScript. Se evaluaron
tres opciones para construir el frontend.

## Opciones consideradas

**a) VanillaJS + fetch**
Sin framework: manipulación directa del DOM, fetch nativo, sin build step
obligatorio.

**b) Angular**
Framework opinionado de Google. Incluye de serie: módulos, servicios con DI,
router, formularios reactivos, HttpClient y CLI propia. Usa TypeScript por
defecto.

**c) React + Vite (opción elegida)**
Librería de UI de Meta. Sin opinión sobre routing, estado ni fetching.
Ecosistema amplio. Se complementa con herramientas elegidas por el desarrollador.

## Decisión

Adoptamos la opción (c): React 19 con Vite, TypeScript, React Router 7
y Axios.

## Razones

1. **Curva de aprendizaje acorde al equipo.** Angular exige aprender DI,
   decoradores, NgModules, RxJS y su propio sistema de formularios antes de
   escribir la primera pantalla útil. React se puede aprender de forma
   incremental: primero componentes, luego hooks, luego estado global si hiciera
   falta.

2. **Sin overhead de framework para un proyecto acotado.** AgroTech tiene
   3 páginas y 6 entidades. Angular está diseñado para aplicaciones enterprise
   con decenas de módulos. Su boilerplate (módulos, servicios, guards, pipes)
   es desproporcionado para esta escala.

3. **VanillaJS no escala con la complejidad de UI reactiva.** Sincronizar
   manualmente el DOM con el estado del servidor (paginación, filtros, modales,
   formularios con validación) produce código frágil y difícil de mantener.
   Los componentes de React resuelven eso de forma declarativa.

4. **Compatibilidad con FSD.** Feature-Sliced Design (ADR-002) requiere slices
   con segmentos `ui/`, `model/`, `api/`. React no impone ninguna estructura de
   carpetas, lo que hace trivial adoptar FSD. Angular tiene su propia convención
   de módulos que entra en conflicto con la organización por feature.

5. **Ecosistema relevante disponible.** Las dependencias elegidas son mínimas y
   directas: `react-router` para navegación, `axios` para HTTP, `lucide-react`
   para iconografía. No fue necesario incorporar librerías de estado global
   porque cada slice gestiona su propio estado con hooks nativos de React.

6. **Tooling moderno sin configuración.** Vite + `@vitejs/plugin-react-swc`
   ofrece HMR instantáneo y builds rápidos sin configuración de Webpack. El
   type-checking con TypeScript 5 y el linting con `eslint-plugin-feature-sliced`
   refuerzan las reglas de FSD automáticamente.

## Stack resultante

| Herramienta                         | Rol                              |
|-------------------------------------|----------------------------------|
| React 19                            | Librería de UI                   |
| Vite 7 + plugin-react-swc          | Build tool y dev server          |
| TypeScript 5                        | Tipado estático                  |
| React Router 7                      | Navegación SPA                   |
| Axios                               | Cliente HTTP                     |
| Lucide React                        | Iconografía                      |
| eslint-plugin-feature-sliced        | Lint de reglas de capas FSD      |
| Stylelint + stylelint-config-standard | Lint de CSS Modules             |

## Consecuencias

+ Menor tiempo de onboarding: React es la librería frontend más documentada
  y con más recursos de aprendizaje disponibles.
+ Flexibilidad total para adoptar FSD sin conflictos con convenciones del
  framework.
+ El bundle final es ligero: sin el runtime de Angular (≈130 kB gzip),
  solo lo que se usa.
- Al no ser un framework completo, cada decisión de arquitectura (routing,
  fetching, estado) recae en el desarrollador; malas elecciones se pagan más
  tarde que con Angular donde ya están tomadas.
- React no tiene opinión sobre formularios; en formularios complejos puede
  requerir una librería adicional (react-hook-form, etc.) que Angular provee
  de serie.
