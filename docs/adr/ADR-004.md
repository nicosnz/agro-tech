# ADR-004: Por qué FastAPI y no Flask

Estado: Aceptada
Fecha: 2026-04-15

## Contexto

`api-vaquero` es el servicio que expone la lógica ganadera de AgroTech vía REST.
Necesita conectarse a tres fuentes de datos externas: PostgreSQL (SQLModel +
asyncpg), Elasticsearch y Redis. El equipo es de 1 persona con conocimiento
previo de Python.

Se evaluó qué microframework usar para construir la API.

## Opciones consideradas

**a) Flask**
Microframework WSGI síncrono. Minimalista, enorme comunidad, muchos tutoriales.
Soporte async añadido en Flask 2.0 (2021) pero como capa opcional sobre un
modelo fundamentalmente síncrono.

**b) FastAPI (opción elegida)**
Framework ASGI async-first construido sobre Starlette y Pydantic. Valida
requests y serializa responses automáticamente a partir de type hints de Python.
Genera documentación OpenAPI sin configuración adicional.

## Decisión

Adoptamos la opción (b): FastAPI 0.135 con uvicorn (incluido en
`fastapi[standard]`).

## Razones

1. **El stack entero es async; Flask no lo es de origen.** Las tres dependencias
   de datos elegidas tienen clientes async: `asyncpg` (PostgreSQL),
   `elasticsearch[async]` y `redis` (async mode). Flask fue diseñado como WSGI
   síncrono; su soporte async en Flask 2+ es un añadido que no puede aprovechar
   un event loop compartido de forma nativa. FastAPI está construido sobre
   Starlette (ASGI) y usa `asyncio` sin fricciones.

2. **Validación automática con Pydantic sin boilerplate.** FastAPI valida
   path params, query params y bodies directamente desde las type hints de
   las funciones. Con Flask habría que integrar marshmallow o WTForms por
   separado y mantener dos esquemas (modelo DB + esquema de validación).

3. **SQLModel unifica modelos de BD y esquemas de API.** SQLModel combina
   SQLAlchemy y Pydantic en la misma clase. El mismo tipo que define la tabla
   PostgreSQL sirve como esquema de request/response, eliminando la duplicación
   modelo-serializer que Flask requiere.

4. **Documentación OpenAPI generada automáticamente.** FastAPI expone `/docs`
   (Swagger UI) y `/redoc` desde el primer endpoint, derivándolos de las
   type hints y docstrings. Con Flask se necesita flask-swagger o flasgger con
   configuración manual. Esto es valioso en un proyecto académico donde la
   documentación de la API es un entregable.

5. **Dependency Injection nativa con `Depends`.** La sesión de base de datos
   se inyecta en cada endpoint vía `Depends(get_db)`. Flask no tiene DI
   integrada; la solución habitual son contextos de aplicación (`g`,
   `current_app`) que son implícitos y difíciles de testear.

6. **`lifespan` para inicialización async de recursos.** FastAPI permite
   inicializar Redis, crear índices de Elasticsearch y sincronizar datos en
   el arranque mediante el context manager `lifespan`, todo en async. Flask
   no tiene un equivalente nativo para inicialización async al inicio de la
   aplicación.

## Stack resultante

| Dependencia               | Rol                                        |
|---------------------------|--------------------------------------------|
| FastAPI 0.135 (standard)  | Framework ASGI + uvicorn + validación      |
| SQLModel 0.0.38           | ORM (SQLAlchemy async) + esquemas Pydantic |
| asyncpg 0.31              | Driver PostgreSQL async                    |
| elasticsearch[async] 8.13 | Cliente Elasticsearch async                |
| redis 7.2                 | Cliente Redis (modo async)                 |

## Consecuencias

+ Un solo model class por entidad sirve tanto para la BD como para la API.
+ Swagger UI disponible en `/docs` sin configuración extra.
+ El event loop único de asyncio gestiona todas las conexiones I/O en paralelo,
  mejorando la throughput sin threads adicionales.
- FastAPI tiene menos tutoriales y recursos en español que Flask, lo que puede
  ralentizar la resolución de problemas específicos.
- SQLModel está en versión < 1.0; algunos comportamientos avanzados de
  SQLAlchemy requieren workarounds al usarlos a través de SQLModel.
